# 베일 오브 이터니티 (Vale of Eternity) 멀티플레이 웹게임 — 작업 메모

Claude Code로 옮겨서 작업을 이어가기 위한 인수인계 메모입니다. 새 세션 시작할 때
"이 파일 읽고 시작해줘"라고 하면 맥락이 빠르게 잡힐 거예요.

## 프로젝트 개요

보드게임 "베일 오브 이터니티"를 실제 룰 그대로 웹 기반 멀티플레이 게임으로 구현 중.
- 백엔드: Node.js + Express 5 + Socket.IO 4 + Mongoose (MongoDB Atlas)
- 프론트엔드: 단일 파일 `index.html` (인라인 CSS + `<script>` 전부 포함)
- 주요 파일: `server.js`(서버), `index.html`(클라이언트), `package.json`
- 카드 데이터: 5속성(🔥불 💧물 ⛰️땅 🍃바람 🐉드래곤) × 각 14~15장, 총 70장. `server.js`의
  `initializeCardDatabase()`에 전부 하드코딩되어 있고 서버 시작 시 MongoDB에 강제 주입됨.

## 게임 구조 (현재 구현 흐름)

1. **방 생성/입장**: `createRoom`/`joinRoom` 소켓 이벤트. 최대 3인.
2. **게임 시작**: 주사위로 선플레이어 순서 결정 (`requestStartGame` → `gameStartSignal`).
3. **DRAFT (드래프트) 단계**: 마켓에 카드 6장 표시, 각 플레이어가 카드에 "찜"(마커)을 박음
   (`claimCard` → `action_buyCard`).
4. **ACTION (액션) 단계**: 자기 차례에 여러 행동(소환/판매/길들이기/제거) 가능, "턴 종료" 버튼을
   직접 눌러야 다음 사람 차례로 넘어감 (`endTurn` → `action_endTurn`). 행동 1개당 턴이 넘어가지
   않도록 주의 — 과거 버그였고 지금은 고쳐짐.
5. **REFRESH (환기) 단계**: 전장에 깔린 환기(REFRESH) 효과 카드들의 발동 순서를 플레이어가
   직접 지정한 뒤(`confirmRefreshOrder`), 그 순서대로 순차 정산.
6. 라운드 종료 시 마켓 6장 전체 교체, 선플레이어 순환.

## 마켓/덱 시스템 (확인 완료, 정상 동작)

- 게임 시작 시 70장 셔플 → 6장은 마켓, 나머지 64장은 `gameMainDeck`(드로우 더미).
- 길들이기(tame)/소환(summon)된 카드는 마켓에서 영구 제거, 다시 등장 안 함.
- 판매(sell)/제거(remove)된 카드만 `discardPile`(폐기더미)에 쌓임.
- 마켓을 새로 채워야 하는데 `gameMainDeck.length < 6`이면, 그 시점에 폐기더미를 덱에 합쳐서
  셔플 후 다시 뽑음. (정확히 "70장 다 소진"이 아니라 "다음 마켓 채우기 부족"이 트리거 — 의도된 설계)

## 지금까지 고친 버그들

1. **찜 마커가 떴다가 사라짐**: 서버에서 `Card.find({})`에 `.lean()` 안 써서 Mongoose 문서의
   `toJSON()`이 스키마 외 필드(`claimedBy`)를 직렬화에서 날려버리던 문제. `.lean()` 추가로 해결.
2. **액션 1개만 해도 턴이 넘어감**: 클라이언트 `sync_turnUpdate` 핸들러가 무조건 `+1` 하던 걸,
   `data.currentTurnOwnerNickname` 기준으로 인덱스를 재계산하도록 수정.
3. **판매한 카드가 마켓에 다시 부활**: 서버 `room.marketCards`를 안 갱신해서 stale 데이터를
   재방송하던 문제. `playerExecutedAction` 핸들러에서 sell/tame 시 `room.marketCards`에서 실제
   splice하도록 수정.
4. **라이벌 상태창 보라색(🟣) 마법석 동기화 안 됨**: `actionSellCard`/`actionTameCard`의
   `playerExecutedAction` emit에 `currentStones`/`currentScore`/`currentSummonZone`이 빠져있어서
   서버가 0으로 기본값 채워 방송하던 문제. emit에 세 필드 추가로 해결.
5. **미믹(312) 환기 효과 닫기 버튼으로 스킵 가능했던 문제**: 미믹 모드 팝업(`openDiscardModal(true, ...)`)
   일 때 "❌ 닫기" 버튼을 `disabled` 처리. `closeDiscardModal(force)`에도 잠금 가드 추가.
6. **패시브 체크 코드 중복 리팩토링**: 헤스티아(마법석 한도)/아그니·해태·물의거인(마법석 가치)
   체크가 9곳에 복붙되어 있던 걸 `getStoneMaxLimit()`, `getStoneValues()` 공용 함수 2개로 통합.
7. **네시(205) 드래곤 판정 버그**: `hasDragon` 체크에 레비아탄/히드라(둘 다 실제로는 💧물 속성,
   🐉드래곤 아님)가 잘못 포함되어 있던 걸 진짜 드래곤 10장(드래곤 알~이터니티) 기준으로 수정.

## 검증 완료된 것들

- 지속 효과(PASSIVE) 11장 전부 구현 확인: 헤스티아, 피닉스, 아그니, 갓파, 해태, 물의 거인,
  트리톤, 가고일, 페가수스, 실프, 히포그리프. 모두 `mySummonZone` 기준 자기 자신 보드만 체크함
  (룰에 맞음 — 패시브는 본인 보드에만 영향).

## 아직 다루지 않은 것 / 알려진 이슈 (이전 리뷰에서 발견, 미해결)

- `server.js`에 MongoDB Atlas 접속 정보(아이디/비밀번호)가 평문으로 박혀 있음 — 보안 이슈.
  나중에 환경변수(.env)로 옮기는 게 좋음.
- 드래곤/속성 판정 로직이 카드별로 `c.type.includes("🐉")` 체크와 카드 이름 하드코딩 리스트가
  혼재되어 있음. 네시 버그처럼 리스트에 잘못된 이름이 들어갈 위험이 있어 — 카드 데이터의
  `type` 필드만으로 판정하도록 점차 정리하는 게 안전함.
- 일부 카드 효과는 placeholder/단순화 상태 (예: 레비아탄의 "지정 플레이어 드래곤 파괴" 같은
  대상 지정형 효과는 "추후 멀티 연동" 주석과 함께 기본 점수만 처리되고 실제 대상 지정/파괴는
  미구현).
- **재접속 기능**: 사용자가 명시적으로 나중으로 미뤄둠. 아직 시작 안 함.
- 카드가 70장 다 랜덤이라 하나하나 수동 테스트하기 어려움 — "일단 전부 구현 → 실제 플레이하며
  버그 잡기" 전략으로 진행 중. 카드별 룰 구현 검토를 하나씩 요청하면서 확인해나가는 패턴.

## 작업 패턴 (Claude Code에서도 이어가면 좋음)

1. 사용자가 카드 이름이나 증상을 던지면, 관련 코드(보통 `confirmRefreshOrder`의 switch case,
   또는 `actionSummonCard`/즉발 효과 분기)를 찾아서 카드 데이터의 `summary`/`effect.action`과
   대조.
2. 실제 게임 룰 기준으로 맞는지 판단 (필요하면 실제 보드게임 룰 기억을 참고).
3. 버그면 원인을 설명하고 고친 뒤, 문법 체크(`node --check`)하고 알려줌.
4. `index.html`만 고치면 새로고침만, `server.js`를 고치면 서버 재시작 필요하다고 매번 명시.
5. 사용자는 반말로 편하게 대화하는 걸 선호함.
